# Quality Bounds Process

## Purpose

Bit-for-bit agreement between RTL and the golden model proves implementation equivalence. It does not prove that the golden model itself has acceptable reconstruction quality. Quality bounds catch errors where the golden model is internally consistent but mathematically wrong, for example a bad window, wrong delay, wrong FFT scaling, or WOLA mistake.

The global quality-bound manifest freezes no-suppression quality limits per vector after the first complete golden-model release.

## Location

The quality-bound file is global:

```text
artifacts/manifests/quality_bounds.json
```

It is keyed by vector name. Do not create separate per-vector quality-bound files unless the spec changes.

## When quality bounds apply

Quality bounds primarily apply to no-suppression behavior, usually `THR2 = 0` or a vector class explicitly marked as no-suppression.

For suppression-active vectors, high reconstruction error may be intentional. Those vectors are still checked for bit-exact artifact matching, row counts, hashes, and metric consistency, but their quality interpretation is different.

## Required fields

At minimum, each vector entry should freeze:

```text
x_in_sha256
y_out_sha256
max_abs_err
sum_sq_err
error_sample_count
```

Recommended additional fields:

```text
sum_abs_err
rmse_decimal
quality_margin_policy
golden_model_version
coeff_manifest_sha256
notes
```

Wide integers must be decimal strings.

## Do not invent numbers

Quality-bound numbers must be generated by the frozen golden model, then reviewed and frozen with documented engineering margin. Do not hand-type placeholder values into a release manifest and call them real.

Placeholder strings such as `<sha256-hex>` or `<decimal-string>` are illegal in frozen artifacts.

## Workflow

### 1. Generate coefficients and vectors

```bash
make coeffs
make vectors
```

### 2. Run the golden model

```bash
make golden
```

### 3. Check artifacts

```bash
make check-artifacts
```

Do not generate quality bounds from unchecked artifacts.

### 4. Generate initial bounds

```bash
make quality-bounds
```

The tool should read `metrics.json` for each no-suppression vector and write candidate bounds.

### 5. Review margin

For each vector, decide whether the bound equals the measured value or includes margin.

Recommended policy for early project phase:

```text
max_abs_err bound = measured max_abs_err unless there is a documented reason for margin
sum_sq_err bound  = measured sum_sq_err unless there is a documented reason for margin
```

Do not add large unexplained margins. Large margins hide real model bugs.

### 6. Freeze manifest

After review, quality bounds are included in `frozen_release_manifest.json`.

## Acceptance check

`artifact_check.py` or a dedicated quality checker should verify:

```text
metrics.max_abs_err <= bounds.max_abs_err
metrics.sum_sq_err  <= bounds.sum_sq_err
metrics.error_sample_count == bounds.error_sample_count
metrics.stream_hashes.x_in_sha256 == bounds.x_in_sha256
metrics.stream_hashes.y_out_sha256 == bounds.y_out_sha256
```

A bound is invalid if it is detached from stream hashes. Without the hashes, the checker cannot know which exact stream the bound describes.

## Why `THR2 = 0` is not exact identity

Phase 1 Haar had exact lossless behavior when threshold was zero. Phase 2 does not inherit that property because it contains quantized windows, quantized twiddles, fixed-point FFT/IFFT rounding, WOLA accumulation, and final output rounding.

Therefore, the correct quality expectation for Phase 2 no-suppression is deterministic near-lossless behavior under frozen bounds.

Real release files must use actual hashes and measured integer fields.

## Failure interpretation

| Failure | Likely meaning |
|---|---|
| `y_out_sha256` mismatch | output artifact changed; rerun release comparison before quality analysis |
| `error_sample_count` mismatch | finite-tail length or delay policy changed |
| `max_abs_err` exceeds bound | possible WOLA, rounding, coefficient, or delay bug |
| `sum_sq_err` exceeds bound | broad quality degradation, maybe scaling or window issue |
| `THR2=0` has masked eligible bins | threshold or protection logic is wrong |

## Updating quality bounds

Quality bounds may change when:

- arithmetic contract changes;
- coefficient tables change;
- vector set changes;
- bug fix changes expected output;
- model version changes intentionally.

Every quality-bound update must include:

1. old metrics;
2. new metrics;
3. old/new stream hashes;
4. reason for change;
5. reviewer signoff in release notes.

Do not update quality bounds merely to make a failing run pass.

## Recommended tests

Quality-bound tooling should include self-tests for:

- missing vector entry;
- hash mismatch;
- measured value below bound;
- measured value equal to bound;
- measured value above bound;
- missing `error_sample_count`;
- wide integer values encoded as JSON numbers instead of strings;
- placeholder strings in frozen files.
